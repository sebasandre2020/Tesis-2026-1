\section{Protocolo LoRa}
LoRa (Long Range) es una tecnología de comunicación inalámbrica diseñada para redes IoT que requieren comunicaciones a largas distancias con un bajo consumo de energía. Su principal ventaja es su capacidad para transmitir datos a distancias que pueden alcanzar los 15 km en áreas rurales y entre 1-3 km en entornos urbanos densos. LoRa opera en bandas de frecuencia ISM no licenciadas, como 433 MHz, 868 MHz (Europa) y 915 MHz (Norteamérica). Su éxito se debe al uso de la modulación Chirp Spread Spectrum (CSS), que codifica la información a través de chirps de frecuencia, permitiendo que LoRa resista interferencias y mantenga la estabilidad de la señal en entornos desafiantes, con una sensibilidad del receptor que puede alcanzar -130 dBm \cite{A_Study_of_LoRa_Long_Range_and_Low_Power_Networks_for_the_Internet_of_Things}.

El rendimiento de LoRa depende de tres parámetros clave: el Spreading Factor (SF), el ancho de banda (BW) y la tasa de corrección de errores (CR). El SF, que varía entre 7 y 12, permite ajustar la relación entre la distancia de transmisión y la tasa de datos. Un SF más alto aumenta el alcance de la señal pero disminuye la velocidad de transmisión. La tasa de símbolos (Rs) se calcula con la fórmula:

\begin{equation}\label{eq
	} Rs = \frac{SF \cdot BW}{2^{SF}} \end{equation}

Por otro lado, la tasa de datos (Rb) también depende del SF y el BW, y se puede calcular con:

\begin{equation}\label{eq
	} Rb = \frac{SF \cdot \left( \frac{4}{4+CR} \right) \cdot BW}{2^{SF}} \cdot 1000 \end{equation}

Dependiendo de la configuración de estos parámetros, LoRa puede ajustarse para optimizar la cobertura o la tasa de bits, según la aplicación IoT específica \cite{A_Survey_of_LoRaWAN_for_IoT_From_Technology_to_Application}.

Para el diseño de una red inalámbrica en un entorno universitario, es fundamental asegurar que la señal pueda transmitirse desde los nodos sensores hasta el gateway. Para ello se evalúa el presupuesto del enlace (Link Budget) y la pérdida de trayectoria en el espacio libre (Free Space Path Loss, FSPL). El FSPL estima la atenuación de la señal que viaja a través del aire sin obstáculos, y se define como:

\begin{equation}\label{eq:fspl}
	FSPL (dB) = 20 \log_{10}(d) + 20 \log_{10}(f) + 32.44
\end{equation}

El presupuesto del enlace determina la máxima pérdida de señal permitida para que el receptor aún pueda decodificar la información, y se calcula mediante la siguiente ecuación:

\begin{equation}\label{eq:link_budget}
	P_{Rx} = P_{Tx} + G_{Tx} - L_{Tx} - FSPL - L_{Rx} + G_{Rx}
\end{equation}

Para garantizar una comunicación confiable, el Link Budget del módulo debe ser mayor o igual a las pérdidas totales en el trayecto, lo que incluye la pérdida en el espacio libre, las pérdidas por obstáculos y un margen de desvanecimiento. Esto se expresa mediante la siguiente inecuación:

\begin{equation}\label{eq:link_budget_inecuacion}
	\text{Link Budget} \geq FSPL + L_{\text{obstáculos}} + \text{Margen}_{\text{desvanecimiento}}
\end{equation}

El cálculo del FSPL y el Link Budget es de vital importancia para el diseño de la red en espacios universitarios, ya que permite estimar teóricamente el alcance máximo y la viabilidad del enlace de comunicación considerando las frecuencias utilizadas (por ejemplo, 915 MHz) y las potencias de transmisión de los módulos LoRa. Esto asegura que la señal tenga la suficiente potencia para superar la atenuación por la distancia y penetrar muros u otros obstáculos típicos de la infraestructura académica.

\section{Descripción de componentes utilizados para el nodo sensor de CO\(_2\)}
\label{sec:nodo_sensor}

A continuación se muestran los componentes electrónicos requeridos para un nodo sensor o nodo final (end node), tal y como se muestra en la Figura \ref{fig: estructura_general_IoT}. Estos nodos recopilan información de su entorno mediante sensores, tranmitirla o recibir instrucciones del gateway, por lo que mantienen una comunicación bidireccional.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.65\linewidth]{images/capitulo2/modulo_lora.png}
	\caption{\centering Módulo LoRa E220-900T30D \cite{modulo_lora}.}
	\label{fig:modulo_lora}
\end{figure}

El módulo E220-900T30D, mostrado en la Figura \ref{fig:modulo_lora}, es un dispositivo que utiliza tecnología LoRa para comunicación en la banda de 868/915 MHz, basado en el chip LLCC68. Ofrece una transmisión eficiente a largas distancias, alcanzando hasta 10 km en condiciones óptimas, y presenta una alta capacidad de resistencia a interferencias. Admite diferentes modos de operación, incluyendo transmisión fija y por difusión, además de contar con funciones de monitoreo y activación por aire, lo que lo hace ideal para aplicaciones que requieren bajo consumo de energía. Este módulo puede funcionar con voltajes de 3.0 a 5.5V, siendo compatible con niveles de 3.3V y 5V en sus interfaces UART. Con un consumo mínimo en modo de suspensión (alrededor de 5 $\mu$A), es adecuado para aplicaciones como automatización del hogar, sistemas de seguridad y control remoto en entornos industriales, soportando temperaturas extremas de $-40$ a $+85$ °C.



\begin{figure}[H]
	\centering
	\includegraphics[width=0.65\linewidth]{images/capitulo2/sensor_co2.png}
	\caption{\centering Sensor de CO\(_2\) MH-Z19B \cite{sensor_co2}.}
	\label{fig:sensor_co2}
\end{figure}

El sensor MH-Z19B es un módulo infrarrojo no dispersivo (NDIR: non dispersive infrared detector) que permite detectar CO\(_2\) en el aire de manera precisa y estable. Su diseño compacto y de bajo consumo incluye compensación de temperatura, lo que lo hace adecuado para aplicaciones como la monitorización de calidad del aire en interiores, sistemas de climatización (HVAC: heating, ventilation and air conditioning) y dispositivos inteligentes. Ofrece múltiples opciones de salida de datos, incluyendo UART y PWM, y es capaz de medir concentraciones de CO\(_2\) en un rango de 0 a 2000 ppm o hasta 5000 ppm, con un tiempo de respuesta inferior a 120 segundos y una vida útil superior a los 5 años.




\begin{figure}[H]
	\centering
	\includegraphics[width=0.65\linewidth]{images/capitulo2/arduino_uno_pins_datasheet.png}
	\caption{Arduino UNO \cite{arduino_uno_datasheet}.}
	\label{fig:arduino}
\end{figure}

El microcontrolador ATMEGA328P, el cual es utilizado en la placa Arduino UNO mostrada en la Figura \ref{fig:arduino}, está basado en una arquitectura AVR de 8 bits y opera a una frecuencia máxima de 16 MHz. Ofrece una memoria Flash de 32 KB, 2 KB de SRAM y 1 KB de EEPROM, junto con múltiples interfaces de comunicación, como UART, SPI e I2C. Además, cuenta con un convertidor ADC de 10 bits y varios temporizadores, lo que lo hace adecuado para aplicaciones de control y monitoreo. Funciona en un rango de voltaje de 1.8 a 5.5 V y admite modos de bajo consumo, como Power-save, Standby y Power-down, lo que lo convierte en una excelente opción para proyectos de electrónica educativa y aplicaciones embebidas que no requieren altas velocidades de procesamiento \cite{arduino_uno_datasheet}.

\section{Descripción de componentes utilizados para el Gateway LoRa}

El gateway o puerta de enlace es el dispositivo encargado de recibir la información transmitida por los múltiples nodos sensores distribuidos en la universidad y reenviarla hacia la plataforma en la nube. Para lograr esta funcionalidad, el gateway requiere capacidades de procesamiento local, recepción LoRa y conectividad a Internet.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.85\linewidth]{images/capitulo2/esp32.png}
	\caption{\centering Microcontrolador ESP32. \cite{ESP32}}
	\label{fig:esp32}
\end{figure}

El núcleo de procesamiento y conectividad del gateway personalizado se basa en el microcontrolador ESP32, mostrado en la Figura \ref{fig:esp32}. Este dispositivo, desarrollado por Espressif Systems, integra conectividad Wi-Fi y Bluetooth (BLE) de bajo consumo, además de contar con un procesador de doble núcleo que puede operar hasta a 240 MHz. Su conectividad Wi-Fi nativa permite que el gateway se conecte a la red local de la universidad para enviar los datos recopilados hacia Azure mediante protocolos estándar como HTTP o MQTT.

Para la recepción de los paquetes provenientes de los nodos sensores, el gateway también emplea el módulo de comunicación LoRa E220-900T30D. El uso del mismo módulo tanto en el transmisor como en el receptor asegura la compatibilidad a nivel físico y de enlace (MAC/P2P). Las especificaciones detalladas y el principio de funcionamiento de este módulo transceptor ya han sido abordados previamente en la subsección del nodo sensor (ver Sección \ref{sec:nodo_sensor}).

\section{Plataforma en la nube y Stack de Software}

Para el despliegue de los servicios del sistema, almacenamiento y visualización, se emplea un ecosistema basado en la plataforma Microsoft Azure. A continuación, se detallan las tecnologías seleccionadas.

\subsection{Microsoft Azure}
Microsoft Azure es una plataforma de computación en la nube de nivel empresarial que ofrece más de 200 productos y servicios en la nube \cite{azure_docs}. Permite construir, ejecutar y administrar aplicaciones en múltiples nubes, de forma local y en el perímetro. En el contexto de sistemas IoT, Azure provee alta disponibilidad, escalabilidad dinámica y robustos mecanismos de seguridad, facilitando la integración fluida entre los datos enviados por el gateway (a través de Wi-Fi) y las interfaces de usuario.

\subsection{Azure Cosmos DB}
Azure Cosmos DB es un servicio de base de datos relacional y NoSQL totalmente administrado para el desarrollo de aplicaciones modernas \cite{cosmosdb_docs}. Ofrece tiempos de respuesta de un solo dígito de milisegundos, escalabilidad automática e instantánea y compatibilidad con APIs de código abierto. Su capacidad para manejar datos no estructurados y de series temporales de manera eficiente lo hace ideal para almacenar las lecturas continuas de CO\(_2\) generadas por la red de sensores.

\subsection{Azure Container Apps y .NET Core}
Para alojar la lógica del backend del sistema, se utiliza Azure Container Apps, un entorno sin servidor (serverless) que permite ejecutar microservicios y aplicaciones en contenedores de manera escalable, sin la sobrecarga de administrar infraestructuras complejas \cite{containerapps_docs}. La lógica del backend, responsable de procesar la telemetría, gestionar las alertas y exponer las APIs REST, se desarrolla utilizando .NET Core, una plataforma de código abierto, multiplataforma y de alto rendimiento que permite crear aplicaciones robustas y eficientes \cite{net_core}.

\subsection{React.js y Azure Static Web Apps}
La interfaz de usuario, encargada de la visualización de los niveles de CO\(_2\) en tiempo real y la gestión del sistema, se desarrolla empleando React.js, una biblioteca de JavaScript de código abierto para construir interfaces de usuario interactivas basadas en componentes \cite{reactjs_docs}. Para el alojamiento y despliegue continuo de este frontend, se utiliza Azure Static Web Apps, un servicio que construye y despliega aplicaciones web de pila completa automáticamente desde un repositorio de código, integrándose nativamente con APIs alojadas y proporcionando entrega de contenido estático distribuido globalmente \cite{staticwebapps_docs}.
